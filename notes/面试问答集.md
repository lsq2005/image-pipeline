## 二、项目深挖

### Q1：你的项目做了什么？一句话概括

> 我用 PySpark 在真实分布式集群上构建了一条图像数据清洗管道——从 Kaggle 取数据，经过清晰度过滤、感知哈希去重、尺寸标准化，最后输出干净的 Parquet 数据供下游 CV 模型训练使用。

---

### Q2：为什么是分布式？500 张图单机也能跑啊？

> 500 张确实单机能跑——我是故意用小数据集验证管道的正确性。但架构设计是按百万级考虑的。
>
> 管道里最贵的操作是自连接去重，复杂度 O(n²)。500 张图只需比较 12 万对，但 100 万张图要比较 5000 亿对——单机内存直接炸。分布式方案是：按哈希前缀分桶，同类图片落入同一分区，只在桶内做自连接，把 O(n²) 降到 O(kn)。
>
> 另外，Spark on YARN 的数据本地性——Executor 和 DataNode 部署在同一节点，计算被调度到数据所在的位置，避免海量数据通过网络传输——这个优势在小数据量下看不出来，但数据量上去后就体现出来了。

---

### Q3：管道各步骤的顺序为什么是「清晰度→去重→Resize」？

> 我遵循「先轻量后重量」原则。清晰度过滤是 O(n)，每个图片过一个拉普拉斯算子就行。自连接去重是 O(n²)，要两两配对做汉明距离。先用 O(n) 操作削减数据量，再让 O(n²) 操作接收更小的 n。
>
> Resize 放在最后是因为它是不可逆的——224×224 丢了原始分辨率信息。在确定一张图要被保留之前，我不对它做破坏性操作。

---

### Q4：阈值 20 和 10 怎么来的？不是拍脑袋？

> 不是查表查出来的。我做了三步：先 `describe()` 看全局分布，再 `orderBy` 升序看最低分样本实际长什么样，最后人眼判断这张图到底该不该丢。对于清晰度，我发现前 5 张明显是光线不足或对焦问题导致的模糊，阈值 20 刚好把它们隔出去。去重阈值 10 我取的是一个保守值——宁可少删（漏掉一些真重复），也不能误杀——对 CV 训练来说，误删好图比保留重复图的代价大。

---

### Q5：为什么感知哈希去重不用 MD5？

> MD5 是文件级哈希——改一个像素，哈希值完全不同。它只能检测「文件是否一模一样」，不能检测「图片内容是否看起来相似」。
>
> 感知哈希的原理是把图片缩小到 8×8、变成灰度、和均值比较生成 64 位二进制指纹。两张图内容相似，指纹就接近，汉明距离小。它对 JPEG 压缩、轻微调色、缩放都有鲁棒性。我选用 pHash（基于 DCT 变换），比 aHash 对缩放更鲁棒，比 dHash 对整体亮度变化更敏感。

---

### Q6：自连接的性能瓶颈在哪？怎么优化？

> 自连接生成 N×(N-1)/2 行，数据倾斜时更严重——如果某几个 hash 前缀特别多，对应分区的数据量远超其他分区，少数 Executor 会成为瓶颈。
>
> 优化方向：用 LSH（局部敏感哈希）先粗筛分桶，只在桶内精确算汉明距离；或者如果数据范围已知，按感知哈希前缀预分组，避免全量 JOIN；另外我在 UDF 里预计算了 pHash，自连接时只比较 64 位字符串，而不是每次重新解码图片。

---

### Q7：你遇到过什么坑？怎么解决的？

> 第一个坑：UDF 在 worker 节点上报 `ModuleNotFoundError: PIL`。原因是 pip install 只装在 master 上，UDF 在 Executor 上执行，worker 上没有 Pillow。解决方法：在每个 worker 节点上统一安装依赖。这让我意识到分布式环境下，运行环境的同步是基础。
>
> 第二个坑：灰度图转灰度报错。我代码里写死了 `cv2.COLOR_RGB2GRAY`，结果数据集里混进了灰度图——1 通道的图不能再转灰度。修法是加了通道数判断：如果是 3 通道彩色图就转，1 通道直接用。
>
> 第三个坑：Kaggle API 下载报 403。不是 API key 的问题——key 只证明"我是谁"，但 Kaggle 还要求我在网页端手动点一下"Accept Terms"，同意了使用条款才能下载。

## 三、HDFS + Spark 核心考点

### Q8：HDFS 的架构是什么？NameNode 和 DataNode 各干什么？

> NameNode 管元数据——哪个文件被切成哪些块、每个块存在哪些 DataNode 上，全部在内存里，是 HDFS 的单点故障点。DataNode 管实际数据——存字节块、定期向 NameNode 心跳汇报。
>
> 我在项目里通过 `hdfs dfsadmin -report` 看到了 2 个 DataNode 在线，通过 NameNode Web UI（9870 端口）浏览了文件系统。我对 HDFS 读写的理解是：读文件时客户端先问 NameNode 要块位置，然后直接和 DataNode 通信——NameNode 不经过数据流，只控制元数据流。

---

### Q9：「移动计算不移动数据」是什么意思？你在项目里怎么体现的？

> 这是 Hadoop/Spark 的核心设计哲学。传统分布式是把数据拉到计算节点上，但大数据场景下数据动辄 TB 级，网络传输成为瓶颈。
>
> HDFS 把数据块分散存在多个节点上，Spark 的调度器根据数据位置把计算任务发到同一个节点上——Executor 和 DataNode 部署在一起，读的是本地磁盘。我在项目里通过 `yarn node -list` 和 Executor hosts 截图验证了这一点：同一个应用的任务分别跑在 worker1 和 worker2 上，各自读各自本地的 HDFS 块。

---

### Q10：DataFrame 和 RDD 的区别？

> RDD 是最底层抽象——没有 Schema，不知道每列是什么类型，像一张没表头的表格。DataFrame 是带 Schema 的 RDD，知道列名和类型，这让 Catalyst 优化器能自动做谓词下推、列剪枝等优化。
>
> Spark 官方推荐优先用 DataFrame，性能更好、代码更简洁。我的项目全程用 DataFrame API，只在需要理解底层执行时有意识地去看 `explain()` 输出的执行计划。

---

### Q11：宽依赖和窄依赖的区别？

> 窄依赖：父 RDD 的每个 Partition 最多被子 RDD 的一个 Partition 使用——比如 `map`、`filter`。不需要 Shuffle，可以流水线执行。
>
> 宽依赖：父 Partition 被多个子 Partition 使用——比如 `groupByKey`、`join`。需要 Shuffle，数据按 key 重新分发。宽依赖是 Stage 的切分边界。
>
> 在我的管道里，清晰度过滤是窄依赖，自连接和 `groupBy` 是宽依赖。如果数据倾斜——比如某种 hash 前缀特别多——宽依赖会触发大量数据进入同一个分区，导致 OOM。

## 四、Hive 考点

### Q12：Hive 是什么？和 MySQL 有什么区别？

> Hive 不是数据库——是一个把 SQL 翻译成分布式任务的工具。数据存 HDFS，计算靠 Spark/MapReduce，Hive 只负责翻译和元数据管理。和 MySQL 的区别：MySQL 自带存储引擎和执行引擎，Hive 什么都不自带，全依赖外部系统。
>
> 我在项目里测试过：用 Hive 的 EXTERNAL TABLE 登记了我管道产出的 Parquet，建表不移动数据，DROP TABLE 不删文件。我跑了一次 DROP TABLE 实验——`SHOW TABLES` 空了，但 HDFS 上 16 个 Parquet 文件一个不少。

---

### Q13：内部表和外部表有什么区别？你的项目用哪个？

> 内部表的数据由 Hive 管理——DROP TABLE 时 HDFS 上的数据文件一起被删除，适用于 Hive 自己的中间结果。外部表只登记路径——删表只删元数据，数据文件不受影响。
>
> 我的项目用外部表。因为数据是我的 PySpark 管道生成的资产，Hive 只是查询入口。我不能让 Hive 的 DROP 操作影响管道产出的 Parquet 文件。

---

### Q14：你的管道数据对应数仓哪一层？

> 我的管道输出是 DWD 层（Data Warehouse Detail）——已清洗的明细数据。上游 ODS 是 Kaggle 下载的原始图片，下游可以基于 DWD 做 DWS 聚合（比如按类别统计猫狗平均清晰度），最终给 CV 模型训练（ADS 层）。

---

### Q15：Hive SQL 和 MySQL SQL 有什么不同？

> HiveQL 支持大多数 SQL 标准语法，但有几个关键差异：`LIMIT` 语义不同（Hive 只是限制输出行数，不保证随机）；Hive 有 `LATERAL VIEW explode()` 用于展开数组/Map 列；Hive 的分区概念天然内建（`PARTITIONED BY`），MySQL 需要手动分表。另外 MySQL 支持行级 UPDATE/DELETE，Hive 是批量追加为主的——这是 HDFS 只支持追加写决定的。

## 五、CV / 图像处理

### Q16：拉普拉斯方差为什么能衡量清晰度？

> 拉普拉斯算子检测的是图像中亮度变化的二阶导数——在边缘处产生较大响应，平坦区域接近零。清晰图边缘多 → 算子响应高 → 方差大；模糊图边缘模糊 → 算子响应低 → 方差小。
>
> 我的实现里有一个关键细节：`cv2.Laplacian` 必须用 `CV_64F`（64 位浮点数），因为拉普拉斯会产生负数（亮→暗的梯度），CV_8U 无符号整型存不了负数。

---

### Q17：感知哈希（pHash）和汉明距离的原理？

> pHash 把图片缩小到 8×8，转灰度，和均值比较生成 64 位二进制指纹。两张图的指纹逐位 XOR，结果里 1 的个数就是汉明距离。距离越小越相似。
>
> 我选 pHash 而不是 MD5 的原因是：MD5 一个像素不同就全不同，pHash 对 JPEG 压缩、轻微调色、缩放都有鲁棒性。阈值定在 10 是保守策略——宁留勿杀。

---

### Q18：为什么 Resize 到 224×224？

> 224×224 是 ImageNet 时代（AlexNet 2012）定下的行业惯例，主流预训练 CNN 模型（ResNet、VGG、EfficientNet）全都硬编码了这个输入尺寸。我用 LANCZOS 插值做缩小——它考虑 64 个邻近像素，对缩小的保留效果最好。quality=95 是 JPEG 的"视觉无损"阈值——近乎无损但文件比 quality=100 小 20-30%。

## 六、场景题

### Q19：如果数据量从 500 张变成 100 万张，你的管道哪里最先崩？怎么改？

> 最先崩的是自连接去重。100 万张图的配对数是 100 万² ÷ 2 ≈ 5000 亿对，任何集群都跑不动。
>
> 改进方案：
> 1. 自连接换成 LSH 分桶——先用局部敏感哈希粗筛，把相似图片分到同一个桶，只在桶内精确算汉明距离，复杂度从 O(n²) 降到 O(kn)
> 2. 逐行 UDF 换成 Pandas UDF，利用 Arrow 批量序列化，性能高 3-100 倍
> 3. 增加分区数，从当前默认分区调大到数百个，让 Shuffle 时每个分区更小
> 4. 如果图片存 HDFS 的话，把 clear 逻辑推到数据读取阶段，不清晰的不进入管道

---

### Q20：如果你除了去重还要做图片分类，你怎么扩展这条管道？

> 管道已经产出了 label（cat/dog）和 224×224 的 JPEG 数据，下游可以直接接分类模型。扩展方案：管道加一个 UDF 调用预训练 ResNet-18 提取特征向量（比如倒数第二层的 512 维），存到 Parquet 新列 `feature_vector`。这条特征加到 pipeline 中间，下游训练分类器时不用重新过一遍 CNN backbone——这就是特征工程的流水线化。

## 七、Agent / 技术视野

### Q21：什么是 Harness？你怎么理解的？

> Harness 是 Prompt + Context 之上的第三层——管「执行循环」。Prompt 管"怎么问得好"，Context 管"给对信息"，Harness 管"AI 做偏了怎么纠"。
>
> 我画了一个类比：CNN 训练是前向传播→损失函数→反向传播→调权重，Harness 是 Agent 输出→独立评估→反馈→调整行为。底层结构完全一样——都是一个「输出→评估→调整→再输出」的闭环。区别是 CNN 调数学参数，Harness 调语义行为策略。
